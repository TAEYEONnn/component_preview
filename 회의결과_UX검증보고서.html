스마트러닝 웹앱 — 회의 결과 화면설계서 전환 가능성 비판적 분석
분석 개요 이 문서는 회의 정리 내용을 화면설계서로 전환하기 전 설계 가능성을 검증한다
기준 1
의사결정이 불완전하거나 모호한 부분
기준 2
사용자 상태(게스트/비수강생/수강생) 기준이 깨지는 지점
기준 3

기준 4
플로우 단위에서 끊기는 지점 (화면 간 연결 문제)

기능은 정의됐지만 UI/상태가 정의되지 않은 부분

1
비밀번호 찾기 — 이메일 링크 방식 vs 인증번호 입력 방식 간 결정 미완료
문제
이메일 링크 클릭 방식과 인증번호 입력 방식 중 어느 것을 사용할지 논의만 됐고 최종 결정 없이 "2트랙 병행"으로만 정리됐다. 링크 방식과 코드 방식은 서버 구현, 화면 수, 만료 처리가 완전히 다른 플로우다.
왜 문제인지
"이메일로 찾기" 탭 선택 후 사용자 경험이 두 방식 중 하나로 고정돼야 설계가 가능하다. 현재 상태에서 설계하면 한 방식으로 그렸다가 다른 방식으로 수정해야 하는 상황이 발생한다.
실제 발생 가능한 UX 리스크
사용자가 이메일 앱으로 넘어갔다가 앱으로 돌아오는 딥링크 처리가 없으면, 링크 클릭 후 브라우저로 열리고 앱은 비밀번호 재설정 완료를 감지하지 못한다. 사용자가 브라우저에서 비밀번호를 바꾼 뒤 앱을 다시 열면 이전 상태로 방치된다.
개선 방향
이메일 탭 선택 시 링크 방식으로 단일화하고, 딥링크 처리 여부(smartlearning://reset-password?token=...) 확정. 토큰 만료 시간(30분)과 만료 후 화면 "링크가 만료됐어요. 다시 요청해 주세요."를 별도 화면으로 설계에 포함.
2
카카오 로그인 사용자 — 비밀번호 찾기 진입 시 플로우 단절
문제
"카카오 로그인 + 이메일 회원가입 병행" 결정됐으나, 비밀번호 찾기 2트랙(이메일/휴대폰)은 이메일·비밀번호 가입자를 전제로 설계됐다. 카카오로 가입한 사용자는 비밀번호 자체가 없다.
왜 문제인지
로그인 화면의 "비밀번호를 잊으셨나요?" 링크를 카카오 가입자가 눌렀을 때 이메일/휴대폰 인증 화면이 뜨지만, 어느 쪽도 의미 있는 결과를 반환하지 못한다. 비밀번호가 없기 때문이다.
실제 발생 가능한 UX 리스크
카카오 가입자가 이메일 인증을 시도해 인증은 성공하지만 비밀번호 재설정 화면에서 아무 처리도 안 된다. 사용자는 앱 버그로 인식하고 이탈한다.
개선 방향
로그인 화면에서 이메일 입력 후 가입 방식을 서버에서 식별. 카카오 가입 이메일로 비밀번호 찾기 시도 시 "카카오 계정으로 가입된 이메일입니다. 카카오 로그인을 이용해 주세요." 안내 및 플로우 중단. 분기 처리 화면을 설계에 추가.
3
카카오 로그인 후 수강생 인증 플로우 — 화면 설계 없음
문제
"카카오 로그인 후 수강생 여부 확인 + 휴대폰 번호 추가 입력 화면 설계"가 액션 아이템으로만 남겨졌다. 카카오 로그인 완료 후 상태 분기(기존 수강생/신규/비수강생)가 설계되지 않았다.
왜 문제인지
카카오 로그인 완료 후 사용자 상태가 세 갈래다: ① 이미 등록된 수강생 → 바로 메인, ② 전화번호 미입력 → 입력 화면, ③ 비수강생 → 비수강생 메인. 세 분기 모두 다른 화면이 필요한데 하나도 정의되지 않았다.
실제 발생 가능한 UX 리스크
카카오 로그인 후 화면 없이 메인만 구현하면, 수강생인데 비수강생 화면을 보거나 그 반대가 된다. 수강생 전용 기능(출석, 빈 강의실 등)이 잘못 노출되거나 차단되는 형태로 사용자에게 직접 노출된다.
개선 방향
카카오 로그인 완료 직후 분기 플로우를 먼저 설계: 신규 → "수강생이신가요?" 선택 → 예: 휴대폰 인증 → 성공 시 수강생 메인 / 실패 시 비수강생 메인. 기존 연동 유저는 저장된 번호로 자동 검증 후 분기. 각 상태의 화면을 별도 페이지로 설계서에 포함.
4
게스트/비수강생/수강생별 드로어 메뉴 구성 — 항목 수준 정의 없음
문제
"게스트/비수강생/수강생에 따라 노출 항목 차등 구성"으로 결정됐지만, 어떤 메뉴가 어느 상태에서 보이는지 항목 단위 정의가 없다. 회의 중 "게스트에게 빈 강의실, 분실물 찾기는 숨김" 언급은 있었으나 전체 목록이 정리되지 않았다.
왜 문제인지
화면 설계서에서 드로어 메뉴를 3개 버전으로 그려야 하는데, 항목 정의가 없으면 개발자와 디자이너가 각자 다르게 해석한다. "서비스를 충분히 보여줘야 한다"는 기획 의도와 "쓸 수 있는 것만 보여준다"는 UX 원칙이 충돌 중이다.
실제 발생 가능한 UX 리스크
비수강생이 드로어에서 "빈 강의실 현황"을 눌렀을 때 로그인 유도 팝업인지, 회색 처리된 채 표시인지, 아예 숨겨지는지 기획이 달라지면 구현이 세 번 바뀐다. 출시 후 권한 처리 버그로 이어진다.
개선 방향
전체 드로어 메뉴 항목을 나열하고, 각 항목마다 게스트 / 비수강생 / 수강생 열을 만들어 표시 / 잠금표시 / 숨김 중 하나로 명시. 잠금 항목 클릭 시 처리 방식(Toast vs 로그인 화면 이동)도 함께 정의. 표 형태로 설계서에 삽입.
5
자주 쓰는 서비스 — 게스트 기본값 및 커스터마이징 여부 미정의
문제
수강생은 2열 아이콘, 게스트/비수강생은 텍스트 메뉴로 방향이 잡혔다. 그러나 게스트에게 보여줄 기본값 서비스 항목이 무엇인지, 커스터마이징 가능 여부가 정의되지 않았다.
왜 문제인지
메인 화면의 자주 쓰는 서비스 영역은 로그인 상태에 따라 다른 항목을 보여줘야 한다. 게스트 기본값이 없으면 해당 컴포넌트가 빈 공간으로 렌더링되거나 수강생용 서비스가 그대로 노출된다.
실제 발생 가능한 UX 리스크
게스트 상태에서 "출석하기" 아이콘이 자주 쓰는 서비스에 표시됐다가 클릭 시 로그인 화면으로 튕기는 경험이 반복되면, 사용자는 앱 자체를 수강생 전용으로 오해하고 이탈한다.
개선 방향
게스트 기본 서비스 항목 확정 (예: 특강·세미나, 자격증 접수, 강의 탐색, 공지사항, FAQ). 수강생 로그인 시 기본값 별도 정의. 커스터마이징 기능을 1차에 포함할지 여부 결정. 두 상태의 컴포넌트를 서로 다른 화면 정의로 설계서에 명시.
6
메인 페이지 수강 예정 강의 — 클릭 동선 정의 불완전
문제
"수강 예정 강의 X개" 텍스트 클릭 시 수강 정보 리스트 페이지의 "수강 예정" 탭으로 이동하기로 결정됐다. 그러나 수강 정보 리스트 페이지의 기본 선택 탭이 어느 탭인지 정의되지 않았다.
왜 문제인지
메인에서 "수강 예정 강의 2개"를 클릭했는데 수강 정보 페이지가 "수강중" 탭으로 열리면 사용자는 클릭한 것과 다른 화면을 보게 된다. 딥링크 수준의 탭 상태 전달이 설계에 반영돼야 한다.
실제 발생 가능한 UX 리스크
사용자가 "수강 예정 강의 2개"를 눌렀는데 "수강중" 목록이 나오면 직접 탭을 찾아 눌러야 한다. 반복되면 메인의 요약 정보를 신뢰하지 않게 된다.
개선 방향
메인 각 수강 상태 요약 항목의 클릭 진입 탭을 명시: "수강 예정 강의 X개" → 수강예정 탭, "수강중" 카드 → 수강중 탭, 메뉴 직접 진입 → 수강중 탭 기본값. 탭 진입 상태를 화면 설계서 경로 정의에 파라미터로 명시 (?tab=upcoming).
7
토스트 메시지 전환 결정 — 적용 범위 미정의
문제
"기존 팝업을 토스트로 대체"하기로 결정됐다. 그러나 어떤 케이스가 토스트이고 어떤 케이스가 모달로 남는지 기준이 없다.
왜 문제인지
토스트는 사용자 행동을 막지 않는 비블로킹 알림이고, 모달은 확인을 요구하는 블로킹 알림이다. 비밀번호 변경 완료는 자동 로그아웃을 수반하므로 토스트가 아닌 모달이 적합하다. 수강 취소처럼 되돌릴 수 없는 행동의 확인도 토스트로는 처리할 수 없다.
실제 발생 가능한 UX 리스크
비밀번호 변경 완료 후 토스트가 1.5초 뜨고 사라지면, 사용자는 변경이 됐는지 인식하지 못한 채 다시 입력을 시도한다. 자동 로그아웃이 진행되는 동안 토스트가 사라져서 왜 로그아웃됐는지 모른다.
개선 방향
기준 정의: 단순 성공 피드백 → Toast / 되돌릴 수 없는 동작 확인 → Modal / 후속 동작이 동반되는 완료(비밀번호 변경→로그아웃) → Modal. 이 기준을 설계서 앞단 공통 UX 가이드로 명시. 각 화면 설계 시 피드백 방식 일관 적용.
8
출석하기 접근성 — 버튼 위치 결정 미완료
문제
수강 정보 리스트에서 바로 출석이 가능하도록 변경하는 제안이 긍정적으로 논의됐으나, 기존 설계(리스트 → 상세 → 출석)와 제안(리스트 → 출석)을 어떻게 병행할지 결정 없이 "조율해보자"로 마무리됐다.
왜 문제인지
출석 행동이 두 진입 경로를 가지면 출석 처리 로직이 두 화면에서 동일하게 구현돼야 한다. 설계 단계에서 진입 경로가 확정되지 않으면 두 경로 모두 그리거나 한쪽만 그리는 모호한 설계서가 만들어진다.
실제 발생 가능한 UX 리스크
리스트에서 출석 버튼을 눌렀는데 수업 시작 30분 전이라 비활성화돼야 하는 상태라면, 버튼이 보이지만 클릭이 안 되거나 에러가 난다. 이 처리를 리스트 뷰에서 어떻게 표시할지 정의되지 않았다.
개선 방향
출석 가능 시간 조건(수업 시작 N분 전~종료 N분 후) 먼저 결정. 조건 충족 시만 리스트에 출석 버튼 활성화, 미충족 시 버튼 숨기거나 비활성화+툴팁 처리. 이 상태 정의를 수강 정보 화면 설계서에 명시. 출석 버튼은 리스트 직접 노출로 단일 경로 확정 권장.
9
드로어 열기 배경 블러 — 웹앱 성능 및 인터랙션 미정의
문제
드로어 오픈 시 배경 블러 처리가 언급됐지만, 블러인지 딤(반투명 레이어)인지, 배경을 터치해서 드로어를 닫는 제스처가 되는지 정의되지 않았다.
왜 문제인지
CSS backdrop-filter: blur()는 저사양 디바이스에서 렌더링 성능에 영향을 준다. 웹앱 특성상 네이티브 앱보다 블러 처리가 무겁게 동작할 수 있다. 배경을 클릭해서 닫는 동작이 설계에 없으면 드로어를 닫는 방법이 X 버튼뿐이어서 모바일 UX 표준에서 벗어난다.
실제 발생 가능한 UX 리스크
사용자가 드로어를 열었다가 닫으려고 배경을 탭하면 아무 반응이 없다. 닫기 버튼을 찾는 과정에서 의도치 않게 메뉴 항목을 선택한다.
개선 방향
블러 대신 반투명 딤 처리(rgba 오버레이, opacity 0.5)로 대체. 딤 영역 탭 시 드로어 닫힘, 스와이프 왼쪽으로 닫힘 인터랙션 명시. 공통 컴포넌트(Drawer) 스펙에 포함해 모든 드로어 사용 화면에 일관 적용.
설계서 전환 시 최대 리스크 3가지 현 상태로 화면설계서를 만들 경우 발생할 수 있는 가장 큰 리스크
1
카카오 로그인 수강생 인증 플로우가 설계에서 통째로 누락된다
카카오 로그인 후 수강생 여부 확인과 휴대폰 번호 입력 화면이 아직 설계되지 않았다. 이 상태에서 나머지 화면 설계서를 만들면 메인 화면 진입 조건이 "이메일 로그인 사용자" 기준으로만 그려진다. 추후 카카오 인증 플로우가 추가될 때 로그인 후 전환 로직, 수강생 상태 판별 분기, 신규/기존 사용자 분리 처리가 설계서 전반을 수정하게 만든다. 수정 범위가 로그인 후 진입하는 모든 화면에 걸친다.
2
드로어 메뉴 구성이 미정의된 채 설계되면 3개 버전을 동시에 수정해야 한다
현재 어떤 메뉴가 어느 상태에서 노출·잠금·숨김 처리되는지 항목 단위 정의가 없다. 설계자가 각자 판단해서 그리면 세 버전의 메뉴 구성이 제각각이 된다. 개발 단계에서 "이 메뉴는 게스트에게 보여야 하나요?" 질문이 반복되고, 확인이 올라올 때마다 기획으로 되돌아간다. 이 불확실성이 해소되지 않은 채 개발에 진입하면 권한 처리 로직이 불일치하는 버그로 이어진다.
3
Toast/Modal 기준 없이 설계하면 피드백 방식이 화면마다 달라진다
"팝업을 토스트로 대체한다"는 결정만 있고 어떤 상황에서 토스트이고 어떤 상황에서 모달인지 기준이 없다. 화면을 여러 명이 나눠 설계할 경우, 동일한 "취소 완료" 피드백이 어떤 화면에서는 토스트, 다른 화면에서는 확인 팝업으로 그려진다. 출시 후에도 화면마다 피드백 UX가 달라 사용자에게 일관성 없는 앱으로 인식된다.
